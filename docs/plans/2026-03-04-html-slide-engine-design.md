# HTML/CSS Slide Engine — Full Custom Presentation Design

**Date**: 2026-03-04
**Status**: Approved
**Goal**: Replace rigid layout templates with AI-generated HTML/CSS slides for world-class, brand-faithful presentations

## Problem

Current slide system uses 11 hardcoded layout types (title, bullets, section, etc.) with a `ThemeConfig` that swaps colors/fonts. When a user says "use Stripe's branding", we can change to navy + purple, but we can't reproduce Stripe's actual design language — their gradient cards, glass effects, specific spacing patterns, rounded UI elements. Slides look generic because they ARE generic.

Slides are fundamentally just HTML + CSS. A world-class Stripe pitch deck uses the same technology as a Stripe landing page. The only reason our slides look like clip art is that we constrain the AI to rigid templates instead of letting it write the actual HTML/CSS.

## Solution

Every slide becomes a self-contained HTML/CSS string generated by the AI. The renderer is a `<div>` with `dangerouslySetInnerHTML`. The AI is the designer — it writes bespoke HTML for every slide, matching whatever brand/style the user wants.

## Data Model

```typescript
interface DeckSlide {
  id: string;
  html: string;                // Full HTML/CSS for the slide (self-contained, 960×540)
  editableFields: {            // AI-declared editable regions
    id: string;                // Unique field ID
    selector: string;          // data-field attribute value
    type: "text" | "heading" | "image" | "list";
    currentValue: string;      // Current text/src content
  }[];
  notes?: string;              // Speaker notes
  imageQuery?: string;         // Unsplash search term for auto-fetching
}
```

### What dies
- `layout` enum (title, bullets, section, etc.)
- `ThemeConfig` interface (17 properties) — style lives in the HTML
- `bullets[]`, `stats[]`, `subtitle`, `content` loose fields — absorbed into `html`
- `theme` column becomes a label/hint only
- `style` jsonb column becomes unnecessary

### What we keep
- `id`, `notes`, `imageQuery`
- `deckops` operation format (CREATE/UPDATE/DELETE/RENAME)
- `theme` as an optional label (not used for rendering)

## Simplified Chat-to-Slides Flow

### Before (4 phases, 2 handoffs)
```
gathering → outlining → [user approves] → generating → viewing
```

### After (3 states, 0 handoffs)
```
idle → generating → viewing
```

**How it works:**

1. **Idle** — User describes what they want in natural language. No rigid interview. AI asks ONE clarifying question max if the request is too vague, otherwise generates immediately.
   - "Make me a pitch deck for Acme" → AI might ask about audience, then generates
   - "Create a 10-slide Stripe-branded deck about Q4 results" → AI generates immediately

2. **Generating** — AI outputs the full deck in one shot. Its chat message includes a brief outline summary so the user sees what's coming. No separate outline approval step.

3. **Viewing** — User sees slides, iterates via chat:
   - "Make slide 3 darker"
   - "Add a pricing slide after slide 5"
   - "Use Apple's design language instead"
   - Content editing via click-to-edit on `data-field` elements

`DeckPhase` type: `"idle" | "generating" | "viewing"`

### Deleted concepts
- `DeckOutlineEditor` component
- `deckOutline` state in the store
- The gathering/outlining/theming phases
- Outline approval flow

## Rendering Architecture

### The HTML Slide Renderer (~50 lines)

```tsx
function SlideRenderer({ slide, scale, onFieldEdit }) {
  return (
    <div
      style={{ width: 960, height: 540, transform: `scale(${scale})`, transformOrigin: 'top left' }}
      dangerouslySetInnerHTML={{ __html: sanitize(slide.html) }}
      onClick={(e) => {
        const field = e.target.closest('[data-field]');
        if (field) onFieldEdit(field.dataset.field);
      }}
    />
  );
}
```

Replaces 600+ lines of hardcoded layout functions.

### Sanitization (DOMPurify)
- **Strip**: `<script>`, `<iframe>`, `<object>`, `<embed>`, event handlers (`onclick`, `onerror`, etc.)
- **Allow**: all CSS, `<style>` blocks, `<img>` with Unsplash/data URLs, SVG, all standard HTML elements
- Use DOMPurify with a permissive allowlist

### Content Editing
1. User clicks text on slide → detect `data-field="xyz"` attribute
2. Show inline contenteditable overlay on that element
3. On blur → update `slide.editableFields[xyz].currentValue` and patch the HTML string
4. For image fields → open Unsplash picker, swap `src`

### Font Loading
- AI declares fonts in HTML via `@import url('https://fonts.googleapis.com/css2?family=...')`
- Extract font URLs from `<style>` block and preload them
- Fonts are part of the HTML — no separate font loading system

## AI Prompt Architecture

### AI persona
"You are a world-class presentation designer who creates bespoke, brand-faithful slides as HTML/CSS."

### HTML/CSS rules for AI
- 960×540 viewport, self-contained per slide
- Inline styles preferred, `<style>` blocks allowed for complex CSS
- `data-field="id"` on every editable element
- Google Fonts only (via @import)
- Images: use `imageQuery` for auto-fetch, or known Unsplash URLs
- No `<script>`, no external resources (except Google Fonts + Unsplash CDN)

### Design principles in prompt
- Whitespace is a design element (40%+ whitespace on most slides)
- Visual hierarchy: one focal point per slide
- Typography scale: max 3 font sizes per slide
- Color: max 3 colors per slide (bg, text, accent)
- Contrast: WCAG AA minimum
- Rhythm: alternate dense and sparse slides

### Brand knowledge
When user says a brand name, AI should reproduce the FULL design language, not just colors:
- **Stripe**: Dark navy #0A2540 bgs, #635BFF accent, Inter, gradient cards with glass/blur effects, rounded corners, subtle grid patterns
- **Apple**: Ultra-minimal, #F5F5F7 light bgs, SF Pro/Inter, massive whitespace, thin text, product imagery
- **Airbnb**: #FF5A5F coral, DM Sans, rounded cards, warm photography, friendly illustrations
- **Notion**: Black on white, EB Garamond serif headings, Inter body, clean borders, emoji icons
- **Linear**: #5E6AD2 purple, dark #0A0A0B bgs, Inter, subtle gradients, technical precision
- etc. (~15 major brands with specific CSS patterns, not just hex values)

### Slide archetypes (guidance, not rigid types)
Title slides, metric dashboards, comparison grids, image heroes, quote layouts, timelines, feature grids, logo walls, pricing tables, team bios — the AI composes freely, these are inspiration not constraints.

### Edit handling
- "Make slide 3 darker" → UPDATE with modified HTML for slide 3 only
- "Add a pricing slide" → UPDATE with new slide inserted
- "Use Apple's style" → UPDATE all slides with redesigned HTML
- Content edits by the user (click-to-edit) are applied client-side without AI

## Export Pipeline

### PDF Export (primary, pixel-perfect)
1. Take each slide's `html` string
2. Wrap in `<html><body style="margin:0;width:960px;height:540px">`
3. Send to existing `/api/export/pdf` → Puppeteer
4. Puppeteer renders at 960×540 → PDF pages
5. **Identical to browser rendering** — what you see is what you get

### PPTX Export (best-effort, image-based)
1. Render each slide's HTML via Puppeteer → screenshot as PNG
2. Embed PNG images into PPTX slides (one image per slide)
3. Not editable in PowerPoint, but visually identical
4. Lower priority — most users want PDF or present in-browser

## Backward Compatibility

Existing decks in the DB have slides as structured objects (with `layout`, `bullets`, etc.).

**Strategy: Lazy migration**
- When opening an old deck, detect old format (has `layout` field, no `html` field)
- Render using a frozen copy of the old SlideRenderer (kept as `LegacySlideRenderer`)
- User can "upgrade" by asking AI to redesign: "Redesign this deck with modern styling"
- No data migration needed, old decks keep working

## Verification Criteria

1. `npm run build` — zero errors
2. New deck creation: "Create a Stripe-branded pitch deck" → generates full HTML/CSS slides matching Stripe's design language (dark navy, gradients, glass effects, Inter font)
3. Content editing: click text on slide → edit inline → HTML updates
4. Style iteration: "Make slide 3 use a lighter background" → AI updates just that slide
5. Full restyle: "Redo this in Apple's minimal style" → all slides regenerated
6. PDF export: pixel-perfect match to browser rendering
7. Old decks: existing structured-format decks render correctly via legacy renderer
8. Performance: slides render in <100ms (just innerHTML + sanitize)
